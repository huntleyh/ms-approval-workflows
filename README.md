# Approvals Demo — Solution Writeup

## Architecture Overview

The solution is a **.NET 8 Razor Pages web application** hosted on **Azure App Service** (Linux B1) backed by a **SQLite database**. Six Azure Logic Apps handle asynchronous notifications, callbacks, expiry checks, and reminders. All approval-workflow intelligence lives in the application and database — Logic Apps are stateless primitives driven by the app.

---

## What Happens to Each Logic App Run?

### LA-1 `la-approval-initiate` — Long-lived per tier

- Triggered by the .NET app (HTTP POST) at the start of each approval tier.
- Sends an **email** and **Teams adaptive card** to the tier's approvers, then **suspends** using an `HttpWebhook` action.
- The run stays in **"Running / Waiting"** state in Azure — indefinitely — until the app POSTs a callback to its unique callback URL with `{"decision": "complete"}`.
- **One LA-1 run fires per tier.** A 3-tier workflow = 3 sequential LA-1 runs, each waiting its own turn.
- When an approver clicks Approve or Reject in the web app, `ApprovalApiController.AdvanceTierAsync` calls the callback URL → that run completes in Azure and the next tier's LA-1 run begins.

### LA-2 `la-approval-callback-relay` — Short-lived relay

- A **stable HTTP endpoint** the web app calls instead of LA-1 directly. LA-1's callback URL changes with every new run, but LA-2's URL is fixed.
- Receives the signal from the app, forwards it to the current LA-1 run's callback. Completes in seconds.
- Decouples the app from needing to store and manage dynamic LA-1 callback URLs directly.

### LA-3 `la-approval-expiry-check` — Stateless recurrence

| Mode | Recurrence |
|------|-----------|
| Demo | Every **1 minute** |
| Production | Every **1 hour** |

- Fires on schedule, calls the app's `/api/approval/process-expiries` endpoint.
- The app queries SQLite for any requests past their expiry date, auto-rejects them, and fires their LA-1 callbacks.
- The LA-3 run **fires and completes immediately** — no waiting state. Each execution is independent.

### LA-4 `la-approval-reminder` — Stateless recurrence

| Mode | Recurrence |
|------|-----------|
| Demo | Every **2 minutes** |
| Production | Every **4 hours** |

- Same pattern as LA-3. Fires on schedule, calls `/api/approval/send-reminders`.
- The app queries for approvers who haven't acted within the reminder window (2 minutes in demo, 24 hours in production) and re-sends Teams/email nudges.
- Short-lived — fires and completes.

### LA-5 `la-approval-reset-handler` — On-demand short-lived

- Triggered manually or via the admin Reset page. Cancels an in-flight workflow.
- Short-lived — fires, acts, completes.

### LA-6 `la-approval-teams-notify` — Short-lived dispatcher

- Called **inline** by LA-1 to post the Teams adaptive card. Runs as a child call during LA-1 execution.
- Completes before LA-1 suspends into its waiting state.

## Logic App Run Lifecycle Summary

```
Workflow submitted
      │
      ▼
LA-1 (Tier 1) ──fires──► sends email + Teams card
      │                           │
      │                    LA-6 completes
      │
      └──► suspends (HttpWebhook) ◄── WAITING in Azure portal
                    │
         [Approver acts in web app]
                    │
                    ▼
         App POSTs callback → LA-1 completes
                    │
                    ▼
         Next tier? → LA-1 (Tier 2) fires...
                    │
         No more tiers? → Request marked Approved/Rejected
```

LA-3 and LA-4 fire independently on their schedules at all times, scanning for overdue/reminder items across all active requests simultaneously.

## How to Cancel and Start Over

There is no in-app Cancel button in the current version. To cancel a live workflow:

1. **Azure Portal** → your resource group → `la-approval-initiate` → **Runs history** → find the run in **"Running"** state → click **Cancel**.
2. In the web app's **Admin / Reset page**, mark the request as `Rejected` or `Cancelled` to unlock the record in SQLite.
3. Submit a new request to start fresh.

> A dedicated `/api/approval/cancel/{requestId}` endpoint could be added to automate both steps in one click when needed for demo or production use.

## How Logic Apps Stay Flexible (Database-Driven)

**LA-1 knows nothing about your approval hierarchy.** When the app triggers it, it sends only the inputs for *that specific tier* in the HTTP body:

```json
{
  "tierNumber": 2,
  "tierName": "Senior Review",
  "approverEmails": ["alice@contoso.com", "bob@contoso.com"],
  "documentUrl": "https://...",
  "callbackUrl": "<auto-generated per run>"
}
```

All the intelligence lives in **SQLite** and is read by `AdvanceTierAsync`:

| What lives in the DB | What LA-1 does |
|---|---|
| How many tiers exist | Doesn't know — just notifies |
| Who approves each tier | Receives a list, sends to them |
| Expiry per tier | App sets `ExpiryDate`; LA-3 enforces it |
| What happens after approval | App decides; calls LA-1 again if more tiers |
| Reminder thresholds | App reads config; LA-4 just calls the API |

You can add tiers, reorder approvers, change expiry windows, or create entirely new workflow templates — **without touching a single Logic App definition**.

## DemoMode

When `App:DemoMode = true` in `appsettings.json`, all time units are treated as **minutes** instead of hours:

| Setting | Production | Demo |
|---|---|---|
| Default expiry (tiers 1–2) | 72 hours | 30 minutes |
| External tier expiry | 96 hours | 45 minutes |
| Reminder threshold | 24 hours after assignment | 2 minutes after assignment |
| LA-3 expiry check | Every 1 hour | Every 1 minute |
| LA-4 reminder check | Every 4 hours | Every 2 minutes |

This allows the full end-to-end flow — submit, remind, expire, approve, delegate — to be demonstrated in a single sitting.

## Delegation

Any valid email address is accepted for delegation — no whitelist required. The delegatee type (Internal/External) is auto-detected: if the email matches a known user in the seed data it inherits their type; otherwise it defaults to `Internal`. The approval token is regenerated for the new delegatee and the tier progress record is updated in SQLite.
